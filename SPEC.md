# 周报助手 — 需求规格说明书 (SPEC)

## 1. 项目概述

### 1.1 项目名称
AI 周报助手

### 1.2 一句话描述
通过聊天交互每日记录工作，周末 AI 自动汇总生成专业周报的个人效率工具。

### 1.3 目标用户
- **主要用户**：个人开发者，日常在 IDE/终端环境工作
- **次要用户**：可分享链接给同事查看周报（未来支持）

---

## 2. 用户痛点

| # | 痛点 | 说明 |
|---|---|---|
| P1 | 遗忘 | 一周忙碌，到周末想不起干了什么 |
| P2 | 格式整理烦 | 每次排版、分类、措辞花大量时间 |
| P3 | 写得不够好 | 写出来的内容不够专业/亮眼 |
| P4 | 拖延成瘾 | 每周写周报是心理负担，能拖就拖 |
| P5 | 信息来源分散 | 工作散落在 Git、聊天、任务系统里，翻找费劲 |

## 3. 功能需求

### 3.1 核心功能（MVP v1.0 实施）

#### F1 — 每日工作记录（聊天式录入）
- **用户故事**：作为用户，我可以通过聊天对话的方式随时记录工作内容，就像跟朋友聊天一样自然。
- **输入示例**：
  - "今天上午修了登录页的 bug #1234"
  - "下午开了需求评审会，确定了 Q2 迭代范围"
  - "昨天写了支付模块的单元测试"
- **验收标准**：
  - 系统能自动解析自然语言中的日期和内容
  - 工作项被正确存储并关联到对应日期
  - 返回确认消息，告知记录成功

#### F2 — 工作项管理（CRUD）
- **用户故事**：作为用户，我可以查看、修改、删除已经记录的工作项。
- **命令**：
  - `/查看 今天` — 查看今日记录
  - `/查看 本周` — 查看本周所有记录
  - `/查看 全部` — 查看所有历史记录
  - `/修改 <id> <新内容>` — 修改指定工作项
  - `/删除 <id>` — 删除指定工作项
- **验收标准**：
  - 所有 CRUD 操作通过聊天命令完成
  - 操作结果即时反馈

#### F3 — 周报生成
- **用户故事**：作为用户，到了周末我只需一个命令就能生成本周周报，不用手动整理。
- **命令**：
  - `/本周周报` — 生成本周周报
  - `/上周周报` — 生成上周周报
- **验收标准**：
  - 系统汇总指定周期内所有工作项
  - 按分类（开发/会议/文档/沟通/其他）分组展示
  - 输出 Markdown 格式，可直接复制粘贴

#### F4 — AI 润色
- **用户故事**：作为用户，我录入的原始记录可以被 AI 提炼、润色，生成更专业、更结构化的周报。
- **验收标准**：
  - 周报生成时自动调用 AI 进行润色
  - 输出分为「润色版」和「原始汇总版」，用户可选择
  - AI 能合并相似工作项、提炼要点、优化措辞
  - AI 能对工作内容进行自动分类

#### F5 — 每日提醒
- **用户故事**：作为用户，我不需要记住去记录，系统会在每天固定时间提醒我。
- **验收标准**：
  - 每个工作日下午 5:00 自动推送提醒消息
  - 提醒时间可配置
  - 提醒消息出现在聊天界面中

#### F6 — 聊天界面
- **用户故事**：作为用户，我通过一个类似微信的聊天界面与系统交互。
- **验收标准**：
  - 界面外观接近微信聊天风格
  - 支持消息气泡展示
  - 消息即时发送和接收
  - 响应式设计，移动端可用

### 3.2 后续版本功能（v2.0+）

#### F7 — 外部工具自动汇总
- 从 Git 提交记录自动提取工作内容
- 从 TAPD / Jira / Linear 等任务系统拉取已完成事项
- 用户可配置需要聚合的数据源

#### F8 — 分享功能
- 生成周报分享链接
- 可设置密码保护或过期时间
- 分享页面展示格式化的只读周报

#### F9 — 数据看板
- 本周工作分布饼图（开发 vs 会议 vs 文档）
- 每周工作投入趋势折线图
- 常用标签/关键词云

#### F10 — 微信/飞书机器人接入
- 企业微信机器人 webhook
- 飞书机器人
- 个人微信（wechaty 方案，注意合规性）

#### F11 — 多格式导出
- Markdown（默认）
- PDF
- Word
- 纯文本

## 4. 技术决策

### 4.1 整体架构

```
┌─────────────────────────────────┐
│        前端（聊天 UI）            │
│   HTML + CSS + Vanilla JS       │
│   轮询 /chat API                 │
└──────────────┬──────────────────┘
               │ REST API
┌──────────────▼──────────────────┐
│       Spring Boot 后端           │
│  ┌───────────────────────────┐  │
│  │  ChatController           │  │
│  │  ChatService (意图识别)    │  │
│  │  WorkLogService (CRUD)    │  │
│  │  ReportService (汇总)     │  │
│  │  AiService (DeepSeek)     │  │
│  │  ReminderService (定时)   │  │
│  └───────────────────────────┘  │
│         ↓ JPA / Hibernate       │
│  ┌───────────────────────────┐  │
│  │        SQLite              │  │
│  └───────────────────────────┘  │
└─────────────────────────────────┘
               │
┌──────────────▼──────────────────┐
│      DeepSeek API (外部)        │
│  deepseek-chat model            │
└─────────────────────────────────┘
```

### 4.2 技术选型

| 决策项 | 选择 | 理由 |
|---|---|---|
| 编程语言 | Java 17+ | 用户指定 |
| 后端框架 | Spring Boot 3.x | Java 生态标准，开发效率高 |
| 数据库 | SQLite | 本地零配置，适合个人工具 |
| ORM | Spring Data JPA + Hibernate | 与 Spring Boot 深度集成 |
| AI 服务 | DeepSeek API | 用户指定，性价比高 |
| 前端 | Vanilla HTML/CSS/JS | 轻量，无需构建工具 |
| 构建工具 | Maven | Java 生态主流 |

### 4.3 数据库设计

#### WorkLog 表（工作记录）

| 字段 | 类型 | 说明 |
|---|---|---|
| id | INTEGER PK AUTOINCREMENT | 主键 |
| content | TEXT NOT NULL | 工作内容原文 |
| log_date | DATE NOT NULL | 工作日期 |
| category | VARCHAR(50) | 分类（开发/会议/文档/沟通/其他） |
| created_at | DATETIME | 创建时间 |
| deleted | BOOLEAN DEFAULT FALSE | 软删除标记 |

### 4.4 意图识别策略（v1）

采用 **关键字 + 规则匹配**，不引入 LLM 做意图识别以节省成本：

| 用户输入模式 | 意图 | 处理方式 |
|---|---|---|
| 包含 `今天`/`昨天`/`上午`/`下午` 等时间词 + 动作描述 | 记录工作 | 解析时间 + 内容 → 存入 WorkLog |
| `/本周周报` | 生成本周周报 | 汇总本周数据 → DeepSeek 润色 → 输出 |
| `/上周周报` | 生成上周周报 | 汇总上周数据 → 输出 |
| `/查看 ...` | 查询工作项 | 按条件返回工作项列表 |
| `/修改 <id> ...` | 修改工作项 | 更新指定记录 |
| `/删除 <id>` | 删除工作项 | 软删除指定记录 |
| `/帮助` 或 `/help` | 帮助 | 返回命令列表和用法说明 |
| 其他 | 闲聊/未知 | 返回引导信息，提示可用的操作 |

### 4.5 AI 服务集成

- **API Endpoint**: `https://api.deepseek.com/v1/chat/completions`
- **Model**: `deepseek-chat`
- **使用场景**：
  - 周报润色（主要场景）：将散乱的工作项整理为结构化、专业的周报
  - 工作内容自动分类（辅助）：分析工作内容并打上分类标签
- **Prompt 设计原则**：
  - 保留工作内容的真实细节，不编造
  - 合并相似工作项，去除冗余
  - 使用专业但自然的职场语言
  - 输出格式：分点列出的 Markdown

### 4.6 部署方式

- **v1**: 本地运行 `java -jar` 或 `mvn spring-boot:run`
- **v2+**: 可考虑部署到云服务器，支持公网访问

## 5. 非功能需求

### 5.1 性能
- 聊天消息响应时间 < 500ms（不含 AI 调用）
- AI 润色响应时间 < 15s

### 5.2 数据安全
- 所有数据存储在本地 SQLite 文件
- AI 调用时仅发送工作内容文本，不附带个人身份信息
- 后续如需云端部署，需支持数据加密

### 5.3 可用性
- 界面简洁，无需培训即可上手
- 帮助命令清晰展示所有可用功能
- 错误提示友好，不显示技术堆栈

### 5.4 可扩展性
- 意图识别设计为可插拔，后续可替换为 LLM 意图识别
- 数据源接口抽象，方便接入 Git/Jira 等
- 通知渠道抽象，方便接入微信/飞书
